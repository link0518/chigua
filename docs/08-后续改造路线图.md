# 08-后续改造路线图

## 1. 背景

当前项目功能已较完整，但关键文件体量持续增长，维护成本和回归风险在上升。

- `server/index.js` 体量较大（约 3200+ 行），集中了配置、路由、业务、风控和运维兜底逻辑
- `components/AdminDashboard.tsx` 体量较大（约 2700+ 行），后台功能耦合度偏高
- `App.tsx` 与 `store/AppContext.tsx` 承担了较多路由和状态职责，后续扩展会继续增加复杂度

## 2. 改造目标

本轮改造目标是“先稳住工程结构，再继续加功能”，分阶段进行，避免一次性大改。

1. 先做无行为变更的模块拆分，降低单文件复杂度
2. 建立清晰的目录边界（配置、设置、路由、业务）
3. 每阶段可独立回滚，确保线上可控

## 3. 非目标（本轮不做）

1. 不改变现有产品交互和业务规则
2. 不引入大规模新依赖或框架迁移
3. 不在同一阶段内同时改数据库模型和前端架构

## 4. 分阶段计划

### 阶段 1（本次实施）

主题：后端入口减负（配置与站点设置逻辑模块化）

改造项：

1. 提取运行时配置模块：统一环境变量加载和运行时配置解析
2. 提取站点设置模块：统一 `app_settings` 读写与春节主题自动生效计算
3. `server/index.js` 仅保留组装与调用，减少重复定义

验收标准：

1. API 行为不变（包括设置读取、公告、后台登录与会话等）
2. `server/index.js` 可读性提升，配置与设置逻辑不再散落
3. 改动文件可通过基础语法检查

### 阶段 2（后续）

主题：后端按领域拆路由

改造项：

1. 拆分为 `routes/public`、`routes/admin`、`routes/system`
2. 复杂 SQL/事务逻辑下沉到 service/repository 层
3. 统一错误返回与参数校验入口

### 阶段 3（后续）

主题：前端路由与状态解耦

改造项：

1. 从手写路径判断迁移到标准路由管理
2. 拆分 `App.tsx` 视图编排职责
3. 将网络状态管理与 UI 局部状态进一步分离

### 阶段 4（后续）

主题：质量门禁与自动化

改造项：

1. 增加 `lint`、`type-check`、`test` 脚本
2. 增加最小 CI 工作流（仅覆盖关键路径）
3. 对高风险接口补充最小回归用例

## 5. 风险与止损

主要风险：

1. 模块提取后导入顺序变化导致运行时行为偏差
2. 配置默认值差异导致线上环境表现变化

止损策略：

1. 阶段 1 仅做无行为变更重构，不触碰业务分支判断
2. 若出现异常，优先回退到单文件实现（按阶段提交可单独回滚）

## 6. 本次交付范围说明

本次按照“先文档后改造”执行：

1. 先完成本文件（改造路线图）
2. 立即实施阶段 1 的代码改造
3. 输出本次改动清单与下一阶段建议
