# 02｜架构总览

本项目是“前后端同仓”的单体应用：

- 前端：Vite + React + TypeScript（入口 `src/index.tsx`，页面/组件位于 `src/components/`）
- 后端：Express（入口 `server/index.js`）
- 数据：SQLite（由 `server/db.js` 初始化并持久化到 `server/data/app.db`）

## 1. 运行时端口与流量路径

- 后端 API：默认 `4395`（环境变量 `PORT`）
- 前端 Dev：默认 `4396`（环境变量 `VITE_PORT`，见 `vite.config.ts`）

开发模式下：

- 浏览器访问 `http://localhost:4396`
- 前端请求统一走 `/api/*`
- Vite proxy 将 `/api/*` 转发到 `http://localhost:${PORT}`（见 `vite.config.ts`）

生产模式下常见做法：

- `npm run build` 生成 `dist/`
- 用反向代理（如 Caddy/Nginx）同时：
  - 将 `/api/*`、`/post/*`、`/robots.txt`、`/sitemap.xml` 转发给后端
  - 其余静态资源与 SPA 路由由前端静态站点承接（或由后端 `express.static(DIST_DIR)` 提供）

## 2. 前端结构

### 2.1 入口与路由

- 入口：`src/index.tsx` 渲染 `App`，并用 `AppProvider` 提供全局状态
- 顶层：`src/App.tsx`
  - 使用 `window.location.pathname` 做“轻路由”
  - 主要路径映射：
    - `/` 与 `/post/:id`：首页/帖子详情（同一视图）
    - `/search`：搜索
    - `/favorites`：收藏
    - `/recruitment`：队伍招募广场、我的招募与密聊列表
    - `/recruitment/chat/:threadId`：招募双方密聊与联系方式交换
    - `/tiancai`：后台入口（管理员）
- 首页视图：`src/components/HomeView.tsx`
  - 首页内部再区分 `focus`（单帖）和 `grid`（列表）两种浏览模式；普通首页按本地存储恢复用户偏好，分享路由 `/post/:id` 始终强制为单帖模式。
  - 单帖模式保留“逐条翻帖 + 临近底部预取”的逻辑；列表模式改为“首屏 20 条 + 底部按钮每次再加载 20 条”，避免移动端和低性能设备一次性渲染过多卡片。
  - 列表卡片正文在新标签页打开 `/post/:id?homeIndex=...`，用于保持当前列表滚动位置，并让详情页的“上一个瓜 / 下一个瓜”沿原列表位置继续浏览；带 `?comment=` 的分享链接只会自动展开一次评论弹窗，关闭后不会因局部状态刷新而重复打开。

### 2.2 API 调用

- `src/api.ts` 统一封装 `fetch('/api' + path)`
  - 自动附带 `credentials: 'include'`（依赖 cookie session）
  - 首次请求后，服务端会补发 `HttpOnly Cookie` `gs_client_id_v2`，作为较稳定的客户端身份
  - 针对部分接口会附带 `X-Client-Fingerprint`，用于 legacy 风控与身份关联
  - 对后台/举报相关请求会附带 `X-CSRF-Token`（登录后由后端下发并写入内存变量）

### 2.3 状态管理

- `src/store/AppContext.tsx` 负责组装状态与业务动作，并提供兼容入口。
- `src/store/` 下的 AppShell、Feed、Content、UserPreferences、Admin、ToastUI 和 AppActions Context 按消费范围暴露状态，减少无关组件重渲染。

## 3. 后端结构

- `server/index.js`：
  - 读取 `.env.local` / `.env`
  - Express 中间件：压缩、JSON body
  - Session：`express-session` + `better-sqlite3-session-store`
  - Turnstile 校验：通过 Cloudflare `siteverify` 进行服务端验证
  - 业务路由：公共 API、后台 API、SEO/静态资源等
- `server/identity-service.js`：
  - 负责补发/解析 `gs_client_id_v2`
  - 维护 Cookie canonical 身份与 legacy 指纹的映射（`identity_aliases`）
  - 为帖子、评论、举报、通知、封禁等场景提供统一的 identity lookup hashes
- 招募领域单独拆分为 `server/routes/public/recruitment-routes.js`、`server/routes/admin/recruitment-routes.js`、`server/services/` 与 `server/repositories/` 下的招募模块；公开目录/招募列表与私有会话分开授权，密聊正文和联系方式使用独立加密域。

## 4. 数据结构概览

详见 `docs/03-数据模型(SQLite).md`，本节只描述大类：

- 内容：`posts`、`comments`
- 互动：`post_reactions_*`、`post_views`、`post_favorites`、`comment_likes`
- 风控：`reports`、`*_report_sessions`、`*_report_fingerprints`、`banned_*`、`identity_aliases`
- 招募：`recruitment_posts`、`recruitment_threads`、`recruitment_messages`、`recruitment_contact_exchanges`、`recruitment_reports` 及通知/审计表
- 后台：`users`（管理员）、`admin_audit_logs`（审计）、`post_edits`（编辑记录）
- 站点：`announcements`、`app_settings`、`vocabulary_words`、统计相关表

## 5. 角色 Wiki 补充

- 前台新增 `/wiki` 与 `/wiki/:slug`，由 `src/components/wiki/WikiView.tsx` 承载，是独立 Nordic 风格页面，不继承主站 `SketchUI`、手绘字体或纸张阴影体系。
- `/wiki` 展示角色画廊，支持搜索、tag 筛选和分页；`/wiki/:slug` 展示名字、tags、记录叙述、相关帖子、分组图片附件和已通过编辑历史。
- Wiki 投稿与编辑都进入后台审核，不会直接公开；后台由 `src/components/AdminWikiPanel.tsx` 接入 `AdminDashboard`。
- 数据层新增 `wiki_entries` 与 `wiki_entry_revisions`。前者保存当前公开快照，后者保存新建和编辑的审核记录。
- 角色 Wiki 的业务字段为 `name`、`narrative`、`tags`、`relatedPostIds` 与 `attachments`；相关帖子只保存帖子 ID，附件按“标题 + 图片 URL 数组”保存，不保存帖子正文快照或图片二进制。
- Wiki 图片复用 `/api/uploads/image?usage=wiki` 上传链路；所有用途共用统一图片上传限流，避免客户端切换 `usage` 绕过额度。公开、历史和后台读取到的都是已持久化图床 URL。
- Wiki 不包含所属机构、能力数值、近期动态等角色档案字段，也不保留对应 JSON 字段。
- 详细规则见 `docs/10-角色Wiki功能.md`。
