# 05｜后台管理功能

后台前端入口为 `/tiancai`（见 `src/App.tsx` 的路由映射），由 `src/components/AdminGate.tsx` 负责登录门禁，登录后进入 `src/components/AdminDashboard.tsx`。

## 1. 启用条件

后端满足以下条件时才会启用后台：

- `SESSION_SECRET` 不为空
- `ADMIN_USERNAME` 不为空
- `ADMIN_PASSWORD` 不为空

若缺失任一项，`AdminGate` 会提示“后台未启用”，需要补齐 `.env.local` 后重启服务。

## 2. 后台主要模块（页面/功能）

后台 UI 以 `AdminDashboard` 为主体，包含多个视图（内部切换）。

### 2.1 概览（overview）

典型内容：

- 统计卡片（今日举报、封禁数、访问/发帖趋势等）
- 图表（按周数据）
- 今日待办会汇总待处理举报、隐藏内容、谣言审核、删除申请、Wiki 审核和未读留言

数据来源：

- `GET /api/admin/stats`

### 2.2 举报处理（reports / processed）

能力点：

- 查看待处理举报列表
- 对举报执行动作：忽略 / 删除 / 禁言 / 封禁（可选带权限与时长）
- 批量处理（若 UI 支持）
- 查看举报详情（不展开接口示例）
- 默认不处理“举报谣言”类记录；该类内容会进入独立的“谣言审核”模块

数据来源：

- `GET /api/reports`
- `POST /api/reports/:id/action`（需要 CSRF）
- `POST /api/admin/reports/batch`（需要 CSRF）

### 2.2.1 删除申请审核（deleteRequests）

能力点：

- 在内容审核下独立查看发帖人提交的删帖申请
- 支持“待处理 / 已处理”切换
- 展示帖子摘要、申请原因、申请人身份与审核信息
- 审核通过后软删除帖子，并通知申请人
- 审核驳回后帖子保持公开，并通知申请人
- 审核说明可选；为空时使用默认通知文案
- 若后台已启用企业微信 Webhook，发帖人提交申请成功后会异步推送待审核提醒

数据来源：

- `GET /api/admin/post-delete-requests`
- `POST /api/admin/post-delete-requests/:id/action`（需要 CSRF）

### 2.3 帖子管理（posts）

能力点（概念上）：

- 列表：按时间/热度/举报数等排序，按状态筛选（正常/已删除）
- 单条操作：删除/恢复、封禁作者等
- 批量操作：对选中帖子执行相同动作
- 进入某帖评论列表（用于评论治理）

数据来源：

- `GET /api/admin/posts`
- `POST /api/admin/posts/:id/action`（需要 CSRF）
- `POST /api/admin/posts/batch`（需要 CSRF）
- `GET /api/admin/posts/:id/comments`

### 2.4 后台发帖/编辑（compose / edit）

能力点：

- 后台发帖（可用于公告、运营贴等）
- 编辑既有帖子（会记录编辑原因，并写入编辑记录/审计）

数据来源：

- `POST /api/admin/posts`（需要 CSRF）
- `POST /api/admin/posts/:id/edit`（需要 CSRF）
- 发布类内容（后台投稿 / 站点公告 / 更新公告）统一复用与前台投稿一致的 Markdown 编辑器，支持工具栏、预览、图片上传、粘贴图片上传与表情插入。
### 2.5 评论治理（posts/:id/comments + comment action）

能力点：

- 在“某帖评论列表”中对单条评论执行动作

数据来源：

- `POST /api/admin/comments/:id/action`（需要 CSRF）

**删除语义（重要）：**

当前后台删除评论是“软删除当前一条”，不级联删除其所有回复：

- 服务端仅执行 `UPDATE comments SET deleted = 1 ... WHERE id = ?`
- 并将 `posts.comments_count` 减 1

若未来需要“删除楼层时连带删除回复树”，必须在业务逻辑中实现级联软删策略，并明确 `comments_count` 的一致性规则。

### 2.6 封禁管理（bans）

能力点：

- 查询封禁列表
- 对 IP / 指纹执行封禁与解封
- 选择封禁权限（post/comment/like/view/site）与时长（1h/1d/7d/永久/自定义）

数据来源：

- `GET /api/admin/bans`
- `POST /api/admin/bans/action`（需要 CSRF）

### 2.7 反馈管理（feedback）

能力点：

- 查看用户反馈
- 查看管理员回复历史
- 回复留言，回复内容会写入历史并通过通知发送给留言提交者
- 标记已读/删除/对相关用户执行封禁

数据来源：

- `GET /api/admin/feedback`
- `POST /api/admin/feedback/:id/action`（需要 CSRF）
- `POST /api/admin/feedback/:id/replies`（需要 CSRF）

### 2.8 审计日志（audit）

能力点：

- 查看后台操作日志（谁在什么时候对什么执行了什么动作）
- 按操作分类、风险等级、目标类型、时间范围和是否填写理由筛选
- 列表展示中文动作标题、风险标签和关键字段变更摘要，详情保留原始 JSON 追溯

数据来源：

- `GET /api/admin/audit-logs`

### 2.9 公告（announcement）

能力点：

- 编辑/预览公告（前台会显示公告入口）
- 清空公告

数据来源：

- `GET /api/admin/announcement`
- `POST /api/admin/announcement`（需要 CSRF）
- `POST /api/admin/announcement/clear`（需要 CSRF）
- 同页新增“更新公告”模块：支持发布多条历史更新公告，并在列表中删除单条记录。
- `GET /api/admin/update-announcements`（管理员）
- `POST /api/admin/update-announcements`（管理员 + CSRF）
- `POST /api/admin/update-announcements/:id/delete`（管理员 + CSRF）
### 2.10 设置（settings）

能力点：

- 开关 Turnstile 等站点级配置
- 配置春节皮肤总开关（仅前台），并显示自动时段状态（农历腊月十六到正月十五）
- 配置各业务限流；帖子、评论和 Wiki 附件共用 `upload` 图片上传限流，默认 `12 次 / 分钟`

数据来源：

- `GET /api/admin/settings`
- `POST /api/admin/settings`（需要 CSRF）

### 2.10.1 商城管理（shop）

后台侧栏入口为 **商城管理**。前台「我的 → 商城」的运营后台，分 Tab：**头像框**、**炫彩昵称**。

权限：复用 `settings:read` / `settings:manage`。

#### 商品线：头像框

- 列表改价/改名/上下架/排序
- 粘贴或导入 Frame Package JSON（schemaVersion 2）
- 与前台同一 FrameRuntime 预览

#### 商品线：炫彩昵称

- 列表展示全部炫彩昵称，支持改价、改名、RGB、**有效期（天）**、上下架
- **直接添加**：填写 id、名称、价格、天数（0=永久），并用 **R/G/B（0～255）** 或取色器指定颜色
- 也支持 `#RRGGBB` 写入（服务端解析）
- 实时预览「匿名用户」流光色

#### 定价与有效期

- 商品字段：`price`（瓜子）+ `durationDays`（天，0=永久）
- 示例：10 瓜子 / 1 天
- 兑换后库存写入 `{ id, expiresAt }`；到期后自动从库存移除并卸下装备，恢复默认头像/昵称色
- 限时商品可再次兑换以**续期**（在剩余时间上叠加天数）

API：

- `GET /api/admin/name-styles`
- `POST /api/admin/name-styles`（CSRF + manage）：body 含 `id,name,price,durationDays,color:{r,g,b}`
- `PATCH /api/admin/name-styles/:id`（可改 `durationDays`）
- 头像框 `PATCH /api/admin/nickname-frames/:id` 同样支持 `durationDays`

前台行为：

- 头像框：新帖 `author_frame_id` 快照
- 炫彩昵称：新帖与新评论 `author_name_style_id` 快照，颜色来自 `name_styles` 表
- 到期后仅影响后续发帖/回复；历史快照不改

### 2.11 敏感词（vocabulary）

能力点：

- 查询敏感词列表（分页）
- 新增、启用/禁用、删除
- 从 `Vocabulary/*.txt` 导入 / 导出

数据来源：

- `GET /api/admin/vocabulary`
- `POST /api/admin/vocabulary`（需要 CSRF）
- `POST /api/admin/vocabulary/:id/toggle`（需要 CSRF）
- `POST /api/admin/vocabulary/:id/delete`（需要 CSRF）
- `POST /api/admin/vocabulary/import`（需要 CSRF）
- `GET /api/admin/vocabulary/export`

### 2.12 谣言审核（rumors）

能力点：

- 单独查看“举报谣言”类记录，不和普通举报混在一起
- 支持按状态筛选：待审核 / 已标记 / 已驳回
- 支持按目标类型筛选：帖子 / 评论
- 支持搜索目标内容、证据、目标 ID
- 展示举报证据摘要，辅助管理员判断
- 支持动作：
  - 判定疑似谣言：将目标标记为 `suspected`
  - 驳回谣言举报：将目标标记为 `rejected`
  - 取消谣言标记：清空当前谣言状态
- 若后台已启用企业微信 Webhook，上述审核动作都会异步推送审核结果提醒
- 前台联动规则：
  - 帖子被标记后，会显示“审核提示：该帖疑似谣言，请谨慎辨别。”
  - 评论被标记后，会显示警示提示，并默认折叠原文，点击后展开

数据来源：

- `GET /api/admin/rumors`
- `POST /api/admin/rumors/:targetType/:targetId/action`（需要 CSRF）

### 2.13 招募治理（recruitment）

能力点：

- 按状态和目标类型查看招募举报。
- 填写审计理由后，查看举报提交时明确绑定的有限消息证据；联系方式只有在双方已解锁后才能被举报，后台读取举报提交时保存的加密快照。
- 对举报执行忽略、处理完成或封禁；封禁同时要求 `recruitment:manage` 与 `user_safety:manage`。
- 对招募执行移除/恢复，对会话执行锁定/解锁，对消息和联系方式执行移除/恢复；每个资源动作都必须在 body 携带 `reportId`，服务端校验举报与目标的关联。
- 会话锁定只限制继续写入，不把 `active` / `closed` 业务状态改造成治理状态。
- 招募举报触发的封禁默认仅限制 `recruit` 与 `chat`；`site` 整站封禁必须由管理员显式勾选。
- 封禁会同时覆盖可解析到的 canonical 身份与 legacy 指纹；旧数据只有 legacy 指纹时只写指纹封禁表。
- 不提供全量密聊列表、按会话浏览密聊或全文搜索入口。

数据来源：

- `GET /api/admin/recruitment/reports`
- `POST /api/admin/recruitment/reports/:id/evidence`（需要 CSRF 和审计理由）
- `POST /api/admin/recruitment/reports/:id/action`（需要 CSRF）
- `POST /api/admin/recruitment/posts/:id/action`（需要 CSRF；body 必须含 `reportId`）
- `POST /api/admin/recruitment/threads/:id/action`（需要 CSRF；body 必须含 `reportId`）
- `POST /api/admin/recruitment/messages/:id/action`（需要 CSRF；body 必须含 `reportId`）
- `POST /api/admin/recruitment/contact-exchanges/:id/action`（需要 CSRF；body 必须含 `reportId`）

资源治理不会接受脱离举报链的裸资源 ID。服务端必须在执行事务中校验 `reportId` 与目标类型、目标 ID 的关联，消息还需确认其为举报主消息或已绑定证据；联系方式举报必须具备双方解锁时保存的加密快照。

## 3. 与前端状态的关系

前端 `src/store/AppContext.tsx` 维护底层状态，`src/store/AdminContext.tsx` 对后台页面暴露：

- `adminSession`：是否登录、csrf token、是否禁用后台
- `reports` / `stats` 等后台数据缓存

后台页面在登录后通常会触发加载：

- `loadReports()`：加载举报列表，并受内容审核读取权限控制。
- `loadStats()`：加载后台统计数据，接口会按管理员读取权限隐藏未授权统计项。


## 4. Wiki 审核模块

后台新增 `Wiki 审核` 模块，由 `src/components/AdminWikiPanel.tsx` 承载，并接入 `AdminDashboard` 侧边栏。

### 4.1 审核记录

审核记录区用于处理用户提交的新瓜条和已有瓜条编辑：

- 待审核：展示 `pending` 的 `create` 与 `edit` 请求。
- 已通过：展示已经发布的版本记录。
- 已拒绝：展示被拒绝的投稿和编辑。
- 支持按状态、动作类型和关键词筛选。
- 管理员可以通过或拒绝待审核记录，拒绝时可填写原因。
- 每条审核记录展示多个相关帖的 ID、当前可用状态与摘要；可用帖子可以新标签页打开，失效帖子明确标记为“不可用”。
- 审核记录展示全部附件标题、图片数量和缩略图，点击任意缩略图可在站内图片查看器中浏览该附件组的全部图片。

规则：

- 新瓜条通过后创建公开瓜条，版本号为 `1`。
- 编辑通过后覆盖当前公开瓜条，版本号 `+1`。
- 审核通过时再次校验全部相关帖子；任一关联已失效时整体失败并显示后端错误，不静默移除部分关联。
- 拒绝不会影响当前公开内容，也不会进入公开详情页历史。

### 4.2 瓜条管理

瓜条管理区用于维护当前公开瓜条：

- 查看当前公开瓜条列表。
- 管理员直接新建瓜条。
- 管理员直接编辑瓜条，支持增删最多 5 个相关帖子、维护分组图片附件；编辑后生成包含完整相关资料的已通过版本。
- 附件编辑支持批量选择图片、缩略图预览、删除单图、上传失败后只重试尚未成功的图片，并可在保存前打开整组查看。
- 删除瓜条后，公开列表和详情页不可见。
- 恢复瓜条后，继续展示当前公开快照和已通过历史。

### 4.3 字段限制

后台 Wiki 只管理以下字段：

- 名字
- 记录叙述
- tags
- 相关帖子 ID 数组：最多 5 个，按添加顺序保存并去重
- 分组图片附件：最多 5 组，每组 1 至 3 张、总计最多 10 张，标题最长 60 个字符

图片只通过 `usage=wiki` 上传到既有图床并保存远端 URL；后台不把 `File`、Blob URL 或 Base64 写入瓜条。单图最大 5MB，仅允许 JPEG、PNG、GIF、WebP。

后台不提供也不应新增所属机构、能力数值、近期动态、视觉描述、社会职能等字段入口。

### 4.4 数据来源

- `GET /api/admin/wiki/revisions`
- `POST /api/admin/wiki/revisions/:id/action`
- `GET /api/admin/wiki/entries`
- `POST /api/admin/wiki/entries`
- `POST /api/admin/wiki/entries/:id/edit`
- `POST /api/admin/wiki/entries/:id/action`
