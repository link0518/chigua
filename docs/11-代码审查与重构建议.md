## 必须优先处理的问题

### 1. 高风险：图床 Token 在前端暴露

证据：

- `D:\common\Cursorwork\吃瓜\api.ts:15` 读取 `import.meta.env.VITE_IMGBED_TOKEN`。
- `D:\common\Cursorwork\吃瓜\api.ts:391` 在浏览器请求里设置 `Authorization: Bearer ...`。

原因：`VITE_` 变量会进入前端构建产物。图床 Token 属于服务端密钥，不应下发到浏览器。

建议：新增服务端上传代理接口，由服务端读取图床 Token；前端只把文件传给本站后端。

## 主要 bug / 缺陷风险

### 2. `test:server` 没有覆盖全部测试文件

证据：

- `D:\common\Cursorwork\吃瓜\package.json:18` 只列出 8 个测试文件。
- 当前 `D:\common\Cursorwork\吃瓜\server\tests\` 下实际存在更多测试文件。
- 手动运行全部 `*.test.js` 得到 78 项通过，而 `npm run test:server` 只有 42 项。

影响：新增测试可能不会进入常规验证，回归风险升高。

建议：把脚本改为运行 `server/tests/*.test.js` 的统一入口。

### 3. 前端直接上传图片，缺少服务端统一校验

证据：

- `D:\common\Cursorwork\吃瓜\api.ts:355` 起实现 `uploadImage`。
- `D:\common\Cursorwork\吃瓜\components\CommentModal.tsx`、`MarkdownComposeEditor.tsx` 都直接调用上传能力。

影响：上传大小、类型、频率、审计无法在服务端统一控制；也与 Token 暴露问题绑定。

建议：与问题 1 一并处理，前端上传改走本站后端接口。

## 依赖与配置问题

### 8. 依赖分类不清

观察：部分构建/工具类依赖在 `dependencies` 中，运行依赖和开发依赖边界不够清晰。

建议：保持生产运行需要的包在 `dependencies`，构建、测试、脚本工具放入 `devDependencies`；调整前先确认部署环境是否执行 `npm install --production`。

## 推荐重构顺序

1. 修复上传密钥暴露：新增服务端上传代理，移除前端 `VITE_IMGBED_TOKEN` 使用。
2. 修正 `test:server`：覆盖全部服务端测试文件。
3. 提取 `buildIdentityMatch` 等服务端公共工具。
4. 拆分 `AdminDashboard.tsx`。
6. 提取前端分享复制工具。
